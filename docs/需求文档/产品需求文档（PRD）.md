# 实验室多端Web库存管理系统 - 产品需求文档（PRD）

## 1. 产品背景与目标

### 1.1 产品背景
当前生物实验室（主要研究免疫方向）的库存管理（如引物、质粒、试剂等）缺乏一个统一、灵活的系统，导致库存信息不清晰、管理效率低下。实验室需要一个支持自定义表格结构的Web系统，以便根据实验室的实际需求灵活管理不同类型的库存信息。

### 1.2 产品目标
- 为生物实验室提供一个灵活、易用的多端Web库存管理系统
- 支持自定义表格结构，满足不同类型库存的管理需求
- 实现库存的高效管理和查询，提升实验室库存管理的规范化和效率
- 支持多端访问，确保实验室成员在不同设备上都能便捷使用
- 提供完整的用户角色和权限控制，确保数据安全

### 1.3 成功指标
- 系统上线时间：5天内完成开发和部署
- 用户满意度：实验室成员对系统的易用性和功能性给予积极反馈
- 系统稳定性：基本稳定运行，无严重故障
- 功能完整性：实现所有核心功能，满足用户基本需求

## 2. 用户画像与使用场景

### 2.1 用户画像

#### 2.1.1 管理员（Admin）
- **角色描述**：实验室负责人或指定的系统管理员
- **核心需求**：
  - 全面管理系统配置和用户权限
  - 定义和管理库存表格结构
  - 执行所有数据操作（添加、修改、删除等）
  - 查看系统操作日志和统计报表

#### 2.1.2 组长（Leader）
- **角色描述**：实验室小组负责人
- **核心需求**：
  - 查看所有表格结构
  - 执行数据操作（添加、修改、删除）
  - 查看报表和统计信息
  - 管理小组内的库存数据

#### 2.1.3 组员（Member）
- **角色描述**：实验人员
- **核心需求**：
  - 查看已定义的表格结构
  - 上传/添加库存数据
  - 查询库存数据
  - 不能修改或删除数据

### 2.2 使用场景

#### 2.2.1 场景一：管理员定义库存表格结构
**场景描述**：实验室需要管理新类型的库存（如新型试剂），管理员需要定义新的表格结构。
**用户流程**：
1. 管理员登录系统
2. 进入表格定义模块
3. 输入总表名（如"新型试剂库存表"）
4. 添加列信息（如"试剂名称"、"规格"、"数量"、"存放位置"等）
5. 保存表格定义
6. 新表格结构显示在已定义表格列表中

#### 2.2.2 场景二：组员添加库存数据
**场景描述**：实验人员收到一批新的引物，需要将其信息录入系统。
**用户流程**：
1. 组员登录系统
2. 在侧边栏选择"引物库存表"
3. 点击"添加数据"按钮
4. 填写引物信息（时间、姓名、引物名称、数量、位置等）
5. 提交表单
6. 数据成功添加到表格中

#### 2.2.3 场景三：组长查询和导出库存数据
**场景描述**：组长需要查询本月的质粒库存情况并导出数据进行分析。
**用户流程**：
1. 组长登录系统
2. 在侧边栏选择"质粒库存表"
3. 使用筛选功能设置查询条件（如时间范围为本月）
4. 系统显示符合条件的数据
5. 点击"导出数据"按钮
6. 系统生成并下载CSV格式的数据文件

## 3. 功能模块详细说明

### 3.1 核心功能模块

#### 3.1.1 用户认证与授权模块

**功能描述**：确保系统的安全访问，实现用户登录、注销和权限控制。

**详细功能**：
- 用户登录：输入用户名和密码登录系统
- 用户注销：退出当前登录状态
- 权限控制：基于角色的权限管理（admin/leader/member）
- 密码重置：新用户首次登录强制修改密码，支持密码复杂度验证
- 操作日志：记录用户登录和关键操作日志

**实现要点**：
- 使用JWT进行身份认证
- 密码使用bcrypt加密存储
- 新用户默认密码为"000000"，首次登录必须修改
- 密码复杂度要求：8位以上，包含大小写字母、数字和特殊字符

#### 3.1.2 表格结构管理模块

**功能描述**：允许管理员定义和管理库存表格的结构。

**详细功能**：
- 创建表格结构：定义总表名和列信息（列名、数据类型）
- 查看表格列表：显示所有已定义的表格结构
- 更新表格结构：修改已定义的表格名称和列信息
- 删除表格结构：删除不再需要的表格结构
- 表格搜索：支持按名称搜索表格结构

**实现要点**：
- 所有数据统一使用字符串类型存储
- 至少保留一列，不能删除所有列
- 表格名称和列名不能为空

#### 3.1.3 库存数据管理模块

**功能描述**：管理库存数据的增删改查和批量操作。

**详细功能**：
- 添加数据：单条录入库存数据
- 查询数据：查看表格中的所有数据
- 修改数据：更新已有数据（仅管理员和组长）
- 删除数据：删除不需要的数据（仅管理员和组长）
- 批量导入：支持CSV格式的批量数据导入
- 批量导出：支持CSV格式的数据导出
- 数据筛选：支持按条件筛选数据
- 数据排序：支持按列排序数据
- 数据搜索：支持全局搜索和高级搜索

**实现要点**：
- 导入数据前进行格式验证
- 导出数据包含所有列信息
- 筛选功能支持多种条件组合

#### 3.1.4 报表统计模块

**功能描述**：提供库存统计和报表功能，支持数据可视化。

**详细功能**：
- 库存统计：查看一段时间内的库存消耗报表
- 多维度分析：支持按时间、类型、人员等维度生成报表
- 数据可视化：使用图表展示统计结果
- 报表导出：支持导出统计报表

**实现要点**：
- 支持自定义统计时间范围
- 图表类型包括柱状图、折线图等
- 报表数据可导出为CSV或Excel格式

### 3.2 辅助功能模块

#### 3.2.1 用户管理模块

**功能描述**：管理系统用户和角色权限。

**详细功能**：
- 添加用户：创建新用户并分配角色
- 查看用户列表：显示所有系统用户
- 更新用户信息：修改用户密码和角色
- 删除用户：移除不再需要的用户
- 用户搜索：支持按用户名搜索用户

**实现要点**：
- 只有管理员可以添加、修改和删除用户
- 用户角色分为admin、leader、member三种

#### 3.2.2 系统设置模块

**功能描述**：提供系统级别的设置和配置。

**详细功能**：
- 系统信息：查看和修改系统名称和版本信息
- 数据库备份：备份系统数据库
- 数据库恢复：从备份文件恢复数据库
- 日志管理：查看和清理系统操作日志

**实现要点**：
- 只有管理员可以访问系统设置
- 备份文件自动命名并包含时间戳
- 日志清理支持按保留天数设置

## 4. 交互流程设计

### 4.1 用户登录流程

```
用户访问登录页面 → 输入用户名和密码 → 点击登录按钮 → 系统验证身份 →
├─ 验证成功 →
│  ├─ 检查是否为首次登录或使用默认密码 →
│  │  ├─ 是 → 强制跳转到密码重置页面 → 完成密码重置 → 进入系统主界面
│  │  └─ 否 → 直接进入系统主界面
│  └─ 记录登录日志
└─ 验证失败 → 显示错误提示 → 重新输入用户名和密码
```

### 4.2 表格结构定义流程

```
管理员登录系统 → 进入表格定义模块 → 输入总表名 → 添加列信息 →
├─ 点击"添加列"按钮 → 新增一列 → 继续编辑列信息
└─ 点击"保存表格"按钮 → 系统验证数据 →
   ├─ 验证成功 → 保存表格结构 → 更新已定义表格列表 → 显示成功提示
   └─ 验证失败 → 显示错误提示 → 重新编辑表格信息
```

### 4.3 库存数据添加流程

```
用户登录系统 → 在侧边栏选择表格 → 点击"添加数据"按钮 → 填写数据表单 →
├─ 点击"保存"按钮 → 系统验证数据 →
│  ├─ 验证成功 → 保存数据 → 更新数据表格 → 显示成功提示
│  └─ 验证失败 → 显示错误提示 → 重新编辑数据
└─ 点击"取消"按钮 → 关闭表单
```

### 4.4 数据导入流程

```
用户登录系统 → 在侧边栏选择表格 → 点击"导入数据"按钮 → 选择CSV文件 →
├─ 点击"开始导入"按钮 → 系统验证文件格式 →
│  ├─ 验证成功 → 开始导入数据 → 显示导入进度 → 导入完成 → 显示导入结果（成功/失败数量）
│  └─ 验证失败 → 显示错误提示 → 重新选择文件
└─ 点击"取消"按钮 → 关闭导入窗口
```

## 5. 界面原型说明

### 5.1 设计原则

- **简洁性**：界面设计简洁明了，减少不必要的元素，突出核心功能
- **一致性**：保持界面元素的一致性，包括颜色、字体、按钮样式等
- **响应式**：采用响应式设计，适配不同终端设备的屏幕尺寸
- **易用性**：操作流程简单直观，降低用户学习成本

### 5.2 主要界面

#### 5.2.1 登录界面
- **功能**：用户登录系统
- **布局**：居中的登录表单，包含用户名、密码输入框和登录按钮
- **设计**：简洁的卡片式设计，突出登录功能
- **元素**：
  - 系统Logo
  - 用户名输入框（带标签和提示）
  - 密码输入框（带标签、提示和显示/隐藏按钮）
  - 登录按钮
  - 错误提示区域

#### 5.2.2 主界面
- **布局结构**：三部分结构
  1. **标题区域**：页面顶部，显示系统名称
  2. **侧边栏区域**：页面左侧，包含表格定义和已定义表格列表
  3. **数据显示区域**：页面右侧，显示选中表格的数据

##### 5.2.2.1 标题区域
- **位置**：页面顶部，宽度100%
- **内容**：系统名称"实验室库存管理系统"
- **设计**：背景色为深蓝色，文字为白色，居中显示，字体大小24px，高度60px
- **元素**：
  - 系统名称
  - 用户信息（右上角显示当前登录用户名和角色）
  - 注销按钮

##### 5.2.2.2 侧边栏区域
- **位置**：页面左侧，宽度250px，高度100%（减去标题区域高度）
- **内容**：
  - **表格定义模块**：
    - 模块标题："表格定义"
    - 总表名输入框：标签"总表名"，输入框（最大长度50字符）
    - 列定义区域：
      - 显示已添加的列，每列包含：
        - 列名输入框：标签"列名"，输入框（最大长度50字符）
        - 数据类型选择框：默认"字符串"类型
        - 删除按钮："×"按钮，点击删除当前列（至少保留一列）

    - "保存表格"按钮："保存表格"按钮，点击保存当前表格定义
  - **已定义表格列表**：

    - 表格名称列表：
      - 每个表名为可点击的链接，点击后数据显示区域加载并显示对应表格的数据

##### 5.2.2.3 数据显示区域
- **位置**：页面右侧，宽度100%（减去侧边栏宽度），高度100%（减去标题区域高度）
- **内容**：
  - **表格信息栏**：
    - 当前选中表格的名称（字体大小18px，加粗）
    - 数据统计信息：显示当前表格的总记录数
  - **操作工具栏**：
    - "添加数据"按钮：点击打开数据添加表单
    - "导入数据"按钮：支持批量导入数据（CSV格式）
    - "导出数据"按钮：支持导出当前表格数据（CSV格式）
    - "刷新数据"按钮：点击重新加载当前表格数据
  - **数据过滤与搜索区**：
    - 全局搜索框：支持按关键字搜索表格中的所有数据
    - 高级过滤：支持按列名设置过滤条件（如：日期范围、等于、包含等）

    - 显示选中表格的所有数据，列名与侧边栏定义的表格结构一致

    - 每行末尾显示操作按钮：
      - 查看详情：点击查看完整数据信息
      - 编辑：点击编辑当前数据（仅管理员和组长可见）
      - 删除：点击删除当前数据（仅管理员和组长可见）
  - **批量操作栏**：
    - 当选择多行数据后显示，支持批量删除操作（仅管理员和组长可见）

## 5. 数据需求与接口定义

### 5.1 数据模型

| 字段名 | 数据类型 | 长度 | 约束 | 描述 |
|--------|----------|------|------|------|
| id | INTEGER | - | PRIMARY KEY, AUTOINCREMENT | 表格结构ID，主键 |
| table_name | VARCHAR | 255 | NOT NULL | 总表名，用于标识不同类型的库存数据 |
| columns | TEXT | - | NOT NULL | 列定义JSON字符串，格式：[{"column_name": "列名", "data_type": "数据类型"}...] |
| created_at | DATETIME | - | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | - | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

#### 5.1.3 库存数据表（inventory_data）

| created_at | DATETIME | - | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | - | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

#### 5.1.4 操作日志表（operation_logs）

| data_id | INTEGER | - | NULL | 涉及的数据ID，外键关联inventory_data表 |
| created_at | DATETIME | - | DEFAULT CURRENT_TIMESTAMP | 操作时间 |

### 5.2 API接口定义

#### 5.2.1 认证接口

- **用户登录**：`POST /api/v1/auth/login`
  - 请求参数：username（用户名）、password（密码）
  - 响应数据：token（JWT令牌）、user（用户信息）

- **用户注销**：`POST /api/v1/auth/logout`
  - 请求头：Authorization: Bearer <token>
  - 响应数据：无

- **获取当前用户信息**：`GET /api/v1/auth/me`
  - 请求头：Authorization: Bearer <token>
  - 响应数据：用户信息

- **重置密码**：`PUT /api/v1/auth/reset-password`
  - 请求头：Authorization: Bearer <token>
  - 请求参数：current_password（当前密码）、new_password（新密码）、confirm_password（确认新密码）
  - 响应数据：无

#### 5.2.2 表格结构管理接口

- **创建表格结构**：`POST /api/v1/tables`
  - 请求头：Authorization: Bearer <token>
  - 请求参数：table_name（总表名）、columns（列定义数组）
  - 响应数据：创建的表格结构信息

- **获取表格结构列表**：`GET /api/v1/tables`
  - 请求头：Authorization: Bearer <token>
  - 请求参数：page（页码）、per_page（每页条数）、search（搜索关键词）
  - 响应数据：表格结构列表及分页信息

- **获取表格结构详情**：`GET /api/v1/tables/{id}`
  - 请求头：Authorization: Bearer <token>
  - 响应数据：表格结构详情

- **更新表格结构**：`PUT /api/v1/tables/{id}`
  - 请求头：Authorization: Bearer <token>

## 6. 非功能需求

### 6.1 性能需求
- **页面加载时间**：页面加载时间不超过2秒，数据操作响应时间不超过1秒
- **并发用户数**：支持同时10-20人在线使用
- **数据处理能力**：能够处理至少10万条库存记录
- **导出性能**：导出1万条数据的时间不超过30秒

### 6.2 安全需求
- **用户认证**：支持用户名和密码登录认证，使用JWT进行身份验证
- **访问控制**：基于角色的权限控制，确保用户只能访问和操作其权限范围内的功能
- **数据安全**：
  - 密码使用bcrypt加密存储
  - 敏感数据在传输过程中使用HTTPS加密
  - 数据库文件设置访问权限，防止未授权访问
- **日志记录**：记录用户登录日志和关键操作日志，便于审计和追溯

### 6.3 可用性需求
- **系统可用性**：系统uptime不低于95%

- **界面友好**：操作界面简洁明了，用户易于学习和使用
- **错误处理**：提供清晰的错误提示信息，帮助用户解决问题
- **帮助文档**：提供系统使用说明和操作指南

### 6.4 兼容性需求
- **浏览器兼容**：支持主流浏览器，包括Chrome（最新版）、Firefox（最新版）、Safari（最新版）、Edge（最新版）

- **操作系统兼容**：支持Windows、macOS、Linux等主流操作系统

### 6.5 可扩展性需求
- **功能扩展**：系统架构支持未来功能扩展，如增加新的表格类型、数据字段等

## 8. 风险评估与应对策略

### 8.1 技术风险

#### 8.1.1 风险描述
自定义表格结构功能实现复杂度较高，可能影响开发进度。

#### 8.1.2 应对策略
- 采用JSON格式存储表格结构和数据，简化数据库设计
- 使用成熟的前端框架实现动态表格渲染
- 分阶段开发，先实现基本功能，再逐步完善高级功能

### 8.2 业务风险

#### 8.2.1 风险描述
用户对系统的使用习惯需要时间适应，可能影响系统使用率。

#### 8.2.2 应对策略
- 提供简洁的操作界面和使用说明
- 组织用户培训，帮助用户快速掌握系统使用方法
- 及时收集用户反馈并进行优化

### 8.3 进度风险

#### 8.3.1 风险描述
开发时间紧张（5天），可能影响功能完整性。

#### 8.3.2 应对策略
- 优先实现核心功能，确保系统可用
- 采用敏捷开发方式，快速迭代
- 合理分配资源，确保关键任务按时完成

### 8.4 数据安全风险

#### 8.4.1 风险描述
系统存储的库存数据可能涉及实验室敏感信息，存在数据泄露风险。

#### 8.4.2 应对策略
- 实现严格的权限控制，确保只有授权用户可以访问数据
- 密码使用bcrypt加密存储
- 记录系统操作日志，便于审计和追溯
